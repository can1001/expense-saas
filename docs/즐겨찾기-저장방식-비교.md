# 즐겨찾기/최근사용 저장 방식 비교

## 현재 구현

- **저장소**: localStorage
- **Storage Key**: `expense-favorite-budgets`, `expense-recent-budgets`
- **범위**: 브라우저별 (사용자 구분 없음)
- **제한**: 즐겨찾기 20개, 최근사용 10개 (LRU)

---

## 방식 비교

### 방식 1: IndexedDB (클라이언트)

```
브라우저 IndexedDB (Dexie.js)
├── favoriteBudgets: [{ id, userId, detail, addedAt }]
└── recentBudgets: [{ id, userId, detail, usedAt }]
```

| 장점 | 단점 |
|------|------|
| 기존 Dexie.js 인프라 활용 | 기기별 저장 (다른 기기에서 동기화 안됨) |
| 오프라인에서도 작동 | 브라우저 데이터 삭제 시 유실 |
| 서버 부하 없음 | 백업/복원 수동 필요 |
| 빠른 응답 속도 | |
| 대용량 저장 가능 (50MB+) | |

**구현 예시**:
```typescript
// lib/db/index.ts 스키마 확장
this.version(2).stores({
  // 기존 테이블
  expenses: 'localId, serverId, status, updatedAt',
  attachments: 'localId, expenseLocalId, status',
  syncQueue: 'id, expenseLocalId, status, priority, createdAt',
  syncMeta: 'key',

  // 신규 테이블
  favoriteBudgets: 'id, userId, addedAt',
  recentBudgets: 'id, userId, usedAt',
});
```

### 방식 2: 서버 DB (사용자별)

```
PostgreSQL (Prisma)
├── UserPreference
│   ├── userId (FK → User)
│   ├── favoriteBudgets: Json
│   └── recentBudgets: Json
```

| 장점 | 단점 |
|------|------|
| 모든 기기에서 동기화 | 서버 요청 필요 |
| 브라우저 삭제해도 유지 | 오프라인 시 사용 불가 |
| 관리자가 데이터 관리 가능 | 서버 부하 증가 |
| 사용자 행동 분석 가능 | API 구현 필요 |

**구현 예시**:
```prisma
// prisma/schema.prisma
model UserPreference {
  id              String   @id @default(cuid())
  userId          String   @unique
  user            User     @relation(fields: [userId], references: [id])
  favoriteBudgets Json     @default("[]")
  recentBudgets   Json     @default("[]")
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
}
```

### 방식 3: 하이브리드 (권장)

```
┌─────────────┐     동기화      ┌─────────────┐
│  IndexedDB  │ ◄──────────────► │  서버 DB    │
│  (로컬)     │                  │  (백업)     │
└─────────────┘                  └─────────────┘
       ↑
   즉시 사용
```

| 장점 | 단점 |
|------|------|
| 오프라인 + 동기화 모두 지원 | 구현 복잡도 높음 |
| 빠른 응답 + 데이터 안전 | 충돌 처리 필요 |
| 기존 오프라인 동기화 인프라 활용 | |

**동작 방식**:
1. 즐겨찾기 추가/삭제 → IndexedDB 즉시 반영
2. 온라인 시 → 서버 동기화 (기존 syncQueue 활용)
3. 로그인 시 → 서버에서 불러와 IndexedDB 업데이트

## 추천

| 상황 | 추천 방식 |
|------|----------|
| 빠른 구현, 단일 기기 사용 | **IndexedDB** |
| 멀티 기기, 데이터 중요 | **서버 DB** |
| 최적의 UX, 장기 운영 | **하이브리드** |

---

## 구현 우선순위

현재 프로젝트는 오프라인 동기화 인프라(Dexie.js)가 이미 구축되어 있으므로:

1. **Phase 1**: IndexedDB로 마이그레이션 (localStorage → IndexedDB)
   - 사용자별 분리 (`userId` 필드 추가)
   - 기존 localStorage 데이터 마이그레이션

2. **Phase 2**: 서버 동기화 추가 (선택)
   - `UserPreference` 모델 추가
   - 동기화 API 구현
   - 충돌 처리 로직
