## 1. 배경 및 결론

여러 조직 유형(교회, 비영리단체, 학교, 기업)이 하나의 SaaS로 지출결의서를 관리할 수 있는지에 대한 검토.

**결론: 단일 시스템이 맞다.**

조직 유형 간 차이(예산 구조, 계정과목, 결재 관행)는 **코드의 차이가 아니라 데이터의 차이**이며, 멀티테넌트 구조에서 템플릿/복제 패턴으로 흡수 가능하다. Salesforce, 더존 등이 업종별로 별도 시스템을 만들지 않는 것과 같은 원리.

## 2. 핵심 설계 원칙

### 2.1 조직 유형(orgType)을 1급 개념으로

`Tenant.orgType` (CHURCH / NONPROFIT / SCHOOL / COMPANY)이 다음 세 가지를 결정한다.

1. **계정과목 템플릿**: 유형별 기본 계정과목 세트를 시드로 준비. 테넌트 생성 시 템플릿을 **참조가 아닌 복제**로 주입. 이후 각 조직이 자유롭게 수정. 템플릿 변경이 기존 테넌트에 영향을 주지 않는 것이 핵심.
2. **용어 레이블 맵**: 같은 필드라도 조직 유형에 따라 다르게 표기. (교회: 부서/직분, 기업: 팀/직급) — i18n 방식의 레이블 JSON으로 처리.
3. **기능 플래그**: 부가세·세금계산서는 기업만, 헌금 연동은 교회만. 코드 분기가 아닌 설정으로 제어.

### 2.2 "복제 후 독립" 원칙

- 템플릿은 청사진, 테넌트 데이터는 복제본.
- `AccountCategory.sourceTemplateId`는 String만 저장하고 FK를 걸지 않는다. 출처 추적은 가능하되, 템플릿을 삭제·개편해도 기존 테넌트 데이터가 깨지지 않는다.

### 2.3 결재선도 설정으로

교회(부서장→재정부장→담임목사)와 기업(팀장→본부장→대표)의 결재 흐름 차이는 orgType 템플릿의 기본값일 뿐, 구조는 동일한 "단계별 승인 체인". 테넌트 어드민이 편집 가능.

### 2.4 어드민 3단 구조

| 계층 | 대상 | 권한 |
|---|---|---|
| 슈퍼 어드민 | 청연컨설팅 | 테넌트 생성/과금/템플릿 관리. 별도 라우트(`/super`) + 별도 인증 경계 |
| 테넌트 어드민 | 각 조직 관리자 | 자기 조직의 계정과목·결재선·사용자 관리 |
| 일반 사용자 | 조직 구성원 | 지출결의서 작성/결재 |

**보안 원칙**: `SuperAdmin`은 `User` 테이블에 `isSuper` 플래그로 넣지 않고 **별도 모델로 완전 분리**. 같은 테이블에 두면 테넌트 스코프 미들웨어에 구멍이 생기기 쉽다. 모델을 나누면 `/super` 라우트에서 별도 세션 검증을 강제할 수 있다.

## 3. 데이터 모델 요약

### 3.1 주요 모델

| 모델 | 성격 | 관리 주체 |
|---|---|---|
| `Tenant` | orgType + settings(Json) 보유 | 슈퍼 어드민 |
| `AccountCategoryTemplate` | 전역 청사진 (`@@unique([orgType, code])`) | 슈퍼 어드민 |
| `AccountCategory` | 테넌트 복제본 (`@@unique([tenantId, code])`) | 테넌트 어드민 |
| `ApprovalLineTemplate` / `ApprovalStepTemplate` | 결재선 청사진 | 슈퍼 어드민 |
| `ApprovalLine` / `ApprovalStep` | 테넌트 결재선 | 테넌트 어드민 |
| `SuperAdmin` | 운영자 계정 (User와 분리) | — |

### 3.2 수입/지출 구분: 데이터 속성 vs 기능 플래그

**구분 원칙**: "수입/지출 구분"은 데이터 속성으로, "수입 모듈 노출 여부"는 기능 플래그로.

```prisma
enum CategoryKind {
  INCOME
  EXPENSE
}

model AccountCategory {
  // ...
  kind CategoryKind @default(EXPENSE)
}
```

- 코드 앞자리(1xxx/5xxx)로 유추하지 않는다. 관례일 뿐이며 조직이 코드 체계를 바꾸면 깨진다. 명시적 필드가 안전.
- 지출결의서 작성 화면은 `kind: EXPENSE`만 필터링.

### 3.3 Tenant.settings 구조 예시

```json
{
  "labels": {
    "department": "부서",
    "position": "직분",
    "budget": "예산(회계연도)"
  },
  "features": {
    "incomeModule": true,
    "budgetModule": true,
    "vat": false,
    "taxInvoice": false,
    "offeringLink": true
  }
}
```

| 플래그 | CHURCH | COMPANY | 의미 |
|---|---|---|---|
| `incomeModule` | true | false | 수입(헌금) 기록·조회 화면 노출 |
| `budgetModule` | true | true | 예산 편성 및 집행률 |
| `vat` | false | true | 부가세 처리 |
| `taxInvoice` | false | true | 세금계산서 증빙 |
| `offeringLink` | true | false | 헌금 연동 |

**운영 원칙**: 플래그를 true→false로 끄더라도 기존 데이터는 삭제하지 않고 화면만 숨긴다. 플래그는 항상 "노출 제어"이지 "데이터 제어"가 아니다.

- Json 하나로 시작하는 이유: 컬럼으로 쪼개면 orgType 추가 시마다 마이그레이션 필요. 쿼리 조건으로 자주 쓰이는 플래그가 생기면 그때 컬럼으로 승격.

## 4. 프로비저닝 흐름

테넌트 생성 시 단일 트랜잭션으로 수행 (부분 생성 방지):

1. `Tenant` 생성 — orgType 기본 settings 복사
2. `AccountCategoryTemplate` → `AccountCategory` 복제 (`sourceTemplateId` 기록)
3. `ApprovalLineTemplate` → `ApprovalLine` 복제 (첫 번째를 기본값으로)
4. 테넌트 어드민 `User` 생성

구현: `provision-tenant.ts` (`provisionTenant()`)

## 5. CHURCH 템플릿 시드 내역

파일: `seed-church-template.ts` — 계정과목 46건 + 결재선 2종. 한국 교회 표준 회계 구조 기반(장로교 노회/총회 상회비 포함).

### 5.1 코드 체계

| 코드대 | 그룹 | 예시 |
|---|---|---|
| 10xx | 헌금수입 | 십일조헌금, 주일헌금, 감사헌금, 절기헌금, 선교헌금, 건축헌금 |
| 19xx | 기타수입 | 이자수입 |
| 51xx | 예배비 | 예배용품비, 성찬비, 강사비, 찬양대운영비 |
| 52xx | 인건비 | 교역자사례비, 직원급여, 사택비, 4대보험료, 퇴직적립금 |
| 53xx | 목회활동비 | 목회활동비, 심방비, 도서연구비 |
| 54xx | 선교비 | 국내/해외선교비, 개척지원비, 선교사후원비 |
| 55xx | 교육비 | 교회학교운영비, 교재비, 수련회비, 제자훈련비 |
| 56xx | 봉사구제비 | 구제비, 경조비, 장학금, 지역섬김비 |
| 57xx | 관리비 | 전기수도가스비, 통신비, 차량유지비, 건물관리비, 수선비, 보험료 |
| 58xx | 행정비 | 사무용품비, 인쇄홍보비, 회의비, 식대 |
| 59xx | 상회비·적립·예비 | 노회상회비, 총회상회비, 건축적립금, 예비비 |

- 수입 계정 포함 이유: 지출결의서 자체에는 쓰이지 않지만, 추후 "예산 대비 집행률" 기능에서 헌금 수입 예산이 기준이 됨. 미리 넣어두면 스키마 변경 없이 확장 가능.
- 시드는 upsert 방식 — 재실행 안전, 템플릿 갱신 가능, 복제된 테넌트 데이터는 불변.

### 5.2 결재선 템플릿

| 이름 | 단계 | 비고 |
|---|---|---|
| 일반 지출 결재선 | 부서장 → 재정부장 → 담임목사 | 기본값 |
| 고액 지출 결재선 | 부서장 → 재정부장 → 담임목사 → 당회서기 | 예: 100만원 이상, 당회 결의 관행 반영 |

## 6. 제품 전략 시사점

- **기업 테넌트**: `incomeModule: false` — 매출은 기존 회계 시스템에서 관리, 지출결의만 사용.
- **교회 테넌트**: `incomeModule: true` — 헌금 수입까지 기록하면 시스템이 사실상 **교회 재정 장부** 역할. 포지셔닝이 "지출결의서 도구"에서 **"교회 재정관리 시스템"**으로 확장되며 가격 책정 근거가 달라진다.

## 7. 마이그레이션 및 파일럿 계획

1. 기존 청연컨설팅 시드의 계정과목 17건 → `AccountCategoryTemplate`(orgType: COMPANY)로 승격
2. `provisionTenant`로 청연컨설팅 재생성 (COMPANY 파일럿)
3. 청연교회를 첫 CHURCH 테넌트로 프로비저닝 (CHURCH 파일럿)
4. 두 유형 나란히 실사용 검증

## 8. Phase 2 후보 과제

- `ApprovalLine.minAmount` 필드 추가 — 금액 임계값 기반 결재선 자동 라우팅
- 예산 편성/집행률 모듈 (`budgetModule`)
- 자주 조회되는 기능 플래그의 컬럼 승격
