# 회원가입 기능 설계

## 1. 요구사항

### 기본 요구사항
- 로그인 페이지(`/login`)에 회원가입 버튼 추가
- 회원가입 페이지(`/signup`) 신규 생성
- 일반 사용자가 직접 계정 생성 가능

### 회원가입 시 입력 항목

| 필드 | 필수 | 설명 |
|------|------|------|
| 아이디 (userid) | O | 로그인용 고유 ID (예: 청연정혜종) |
| 이름 (username) | O | 표시 이름 (예: 정혜종) |
| 비밀번호 (password) | O | 로그인 비밀번호 |
| 비밀번호 확인 | O | 비밀번호 재입력 확인 |
| 부서 (department) | X | 소속 부서 (선택) |

### 역할(Role) 정책
- 회원가입 시 역할은 **`user`(일반 사용자)**로 자동 설정
- 역할 변경은 관리자만 가능 (`/admin/users`)

---

## 2. 현재 시스템 구조

### 기존 구현
- **로그인 API**: `POST /api/auth/login`
- **사용자 생성 API**: `POST /api/users` (관리자용)
- **관리자 사용자 추가 페이지**: `/admin/users/new`

### 기존 API 활용
`POST /api/users` API를 회원가입에서도 사용 가능:
```typescript
// 요청 본문
{
  userid: string;      // 필수
  username: string;    // 필수
  password: string;    // 선택 (미입력 시 기본값 사용)
  role: 'user';        // 고정
  department?: string; // 선택
}
```

### 보안 고려사항
- 현재 `/api/users` POST는 인증 없이 접근 가능
- 회원가입용으로 사용하려면 역할을 `user`로 제한 필요
- 또는 별도의 회원가입 API (`/api/auth/signup`) 생성 고려

---

## 3. 구현 방안

### 방안 A: 기존 API 활용 (간단)
- `/api/users` POST를 그대로 사용
- 클라이언트에서 `role: 'user'`로 고정 전송
- **장점**: 빠른 구현
- **단점**: API 레벨에서 역할 제한 없음

### 방안 B: 회원가입 전용 API 생성 (권장)
- `/api/auth/signup` 신규 생성
- 역할을 `user`로 서버에서 강제 고정
- **장점**: 보안 강화, 명확한 책임 분리
- **단점**: API 추가 필요

**권장: 방안 B**

```
    └── auth/

```

### 4.2 회원가입 API 설계

**엔드포인트**: `POST /api/auth/signup`

**요청**:
```typescript
{
  userid: string;      // 필수, 고유
  username: string;    // 필수
  password: string;    // 필수
  department?: string; // 선택
}
```

**응답 (성공 201)**:
```typescript
{
  id: string;
  userid: string;
  username: string;
  role: 'user';
  department: string | null;
}
```

**응답 (에러)**:
- `400`: 필수 필드 누락 또는 유효성 검증 실패
- `409`: 이미 존재하는 아이디

## 5. 유효성 검증

### 클라이언트 검증
- 아이디: 필수, 공백 제거
- 이름: 필수, 공백 제거
- 비밀번호: 필수, 최소 4자 이상
- 비밀번호 확인: 비밀번호와 일치 여부

### 서버 검증
- 아이디 중복 확인 (409 에러)
- 필수 필드 검증 (400 에러)
- 비밀번호 해싱 (bcrypt)

## 7. 추가 고려사항

### 7.1 회원가입 후 자동 로그인
- 회원가입 성공 후 자동으로 로그인 처리
- 또는 로그인 페이지로 리다이렉트 (메시지 표시)

**권장**: 로그인 페이지로 리다이렉트 + 성공 메시지

### 7.2 비밀번호 정책
- 현재: 특별한 제한 없음
- 권장: 최소 4자 이상 (MVP 수준)

### 7.3 아이디 형식
- 현재 시스템: "청연" + 이름 형태 (예: 청연정혜종)
- 회원가입에서도 동일 형식 권장 (placeholder로 안내)
