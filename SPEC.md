# 지출결의서 복제 기능 스펙

## 1. 목표 (Objective)

### 문제 정의
사용자가 유사한 지출결의서를 반복적으로 작성할 때 동일한 정보를 매번 입력해야 하는 불편함이 있음.

### 해결 방안
기존 지출결의서를 복제하여 새로운 지출결의서를 생성하고, 편집 페이지에서 필요한 부분만 수정할 수 있도록 함.

### 대상 사용자
- 지출결의서를 작성하는 모든 사용자
- 본인이 작성한 지출결의서만 복제 가능

### 성공 기준
- 복제 버튼 클릭 → 새 지출결의서 생성 → 편집 페이지 이동이 3초 이내 완료
- 복제된 지출결의서의 모든 데이터가 정확히 복사됨 (서명, 청구일자 제외)

---

## 2. 핵심 기능 (Features)

### 2.1 복제 대상 데이터

| 항목 | 복제 여부 | 비고 |
|------|:--------:|------|
| 위원회 (committee) | O | |
| 사역팀/부 (department) | O | |
| 세부 항목 (items) | O | 항/목/세목, 적요, 단가, 수량, 금액 |
| 청구인 이름 (applicantName) | O | |
| 청구인 직책 (applicantTitle) | O | |
| 은행명 (bankName) | O | |
| 계좌번호 (accountNumber) | O | |
| 예금주 (accountHolder) | O | |
| 첨부파일 (attachments) | O | 원본 이미지 URL 참조 복사 |
| 청구일자 (requestDate) | X | 오늘 날짜로 설정 |
| 청구팀 (requestTeam) | O | 위원회 + 사역팀에서 자동 파생 |
| 서명 (applicantSignature*) | X | 새로 서명 필요 |
| 지출일자 (expenseDate) | X | null로 초기화 |

### 2.2 복제 조건

- **권한**: 본인이 작성한 지출결의서만 복제 가능
- **상태**: 모든 상태의 지출결의서 복제 가능 (DRAFT, PENDING, APPROVED_*, REJECTED, WITHDRAWN)
- **결과**: 복제된 지출결의서는 항상 `DRAFT` 상태로 생성

### 2.3 UI 위치

1. **지출결의서 상세 페이지** (`/expenses/[id]`)
   - 상단 액션 버튼 영역에 "복제" 버튼 추가
   - 아이콘: Copy 또는 Files (lucide-react)

2. **지출결의서 목록 페이지** (`/expenses`)
   - 모바일: 카드 스와이프 메뉴에 "복제" 옵션 추가
   - 데스크톱: 행 액션 메뉴에 "복제" 옵션 추가

### 2.4 복제 후 동작

1. 새 지출결의서 생성 (DB 저장)
2. 편집 페이지로 자동 이동 (`/expenses/[newId]/edit`)
3. 토스트 메시지: "지출결의서가 복제되었습니다."

## 3. API 설계 (Commands)

### 3.1 복제 API

```
POST /api/expenses/[id]/duplicate
```

**요청**
- Headers: 인증 필요 (현재 로그인 사용자)
- Body: 없음

**응답 (성공)**
```json
{
  "success": true,
  "message": "지출결의서가 복제되었습니다.",
  "expense": {
    "id": "new-expense-id",
    ...
  }
}
```

**응답 (실패)**
```json
{
  "error": "본인이 작성한 지출결의서만 복제할 수 있습니다.",
  "statusCode": 403
}
```

**에러 케이스**
| 상태 코드 | 메시지 |
|----------|--------|
| 401 | 로그인이 필요합니다. |
| 403 | 본인이 작성한 지출결의서만 복제할 수 있습니다. |
| 404 | 지출결의서를 찾을 수 없습니다. |
| 500 | 복제 중 오류가 발생했습니다. |

## 5. 코드 스타일 (Code Style)

### API 패턴
- 기존 `/api/expenses/[id]/*.ts` 패턴 따름
- `handleApiError`, `ApiError` 사용
- `getCurrentUser()` 로 인증 확인

### UI 패턴
- 버튼: `BTN_OUTLINE` 또는 `BTN_SECONDARY` 스타일
- 아이콘: `lucide-react`의 `Copy` 또는 `Files`
- 토스트: 기존 토스트 시스템 사용 (있다면) 또는 alert

### 명명 규칙
- API 함수: `POST` (route.ts 내)
- 클라이언트 함수: `handleDuplicate`, `duplicateExpense`

## 6. 테스트 전략 (Testing Strategy)

### 수동 테스트 체크리스트

- [ ] 본인 지출결의서 복제 성공
- [ ] 타인 지출결의서 복제 시 403 에러
- [ ] 복제된 데이터 정확성 확인 (항목, 금액, 은행정보 등)
- [ ] 청구일자가 오늘 날짜로 설정됨
- [ ] 서명 데이터가 제외됨
- [ ] 상태가 DRAFT로 설정됨
- [ ] 편집 페이지로 자동 이동
- [ ] 첨부파일 복제 확인
- [ ] 모든 상태(DRAFT, PENDING, APPROVED, REJECTED)에서 복제 가능

### 엣지 케이스

- [ ] 로그아웃 상태에서 복제 시도 → 401
- [ ] 존재하지 않는 ID로 복제 시도 → 404
- [ ] 첨부파일 없는 지출결의서 복제
- [ ] 항목이 10개인 지출결의서 복제

## 7. 경계 조건 (Boundaries)

### 항상 해야 할 것 (Always Do)
- 본인 확인 (userId 검증)
- 트랜잭션으로 원자성 보장
- 청구일자를 오늘 날짜로 설정
- 서명 데이터 제외
- 상태를 DRAFT로 초기화

### 확인이 필요한 것 (Ask First)
- 첨부파일 처리 방식 변경 (현재: URL 참조 복사)
- 버튼 디자인/위치 변경

### 절대 하면 안 되는 것 (Never Do)
- 타인의 지출결의서 복제 허용
- 결재선/결재 상태 복제
- 원본 지출결의서 수정/삭제
- 서명 데이터 복제
